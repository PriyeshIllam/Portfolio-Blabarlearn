Designing for the way people actually learn

PromptCraft is an accessibility-first LMS built for neurodivergent learners in Daglig Verksamhet — where one-size-fits-all has never worked.

01 · The Problem

Learning platforms assume a standard learner.
Most learners are not standard.

During my LIA internship at a Daglig Verksamhet organisation in Helsingborg, I observed a consistent friction: existing LMS tools were built around a neurotypical model of learning — fixed pacing, dense information, no customisation. For neurodivergent learners, this created anxiety, disengagement, and over-reliance on facilitators for tasks the platform should have handled.

Design Brief: Build a programming learning platform that adapts to the learner's cognitive and sensory preferences. Reduce facilitator burden. Preserve learner autonomy.

The core HCI challenge: how do you surface the right interface configuration for a learner without making them wade through settings they don't understand? The answer became an onboarding-driven persona system that maps natural preference questions to UI toggle states automatically.

02 · User Research

Four personas. Real needs. Verified coverage.

Rather than decorative personas, each profile maps directly to UI toggle states, onboarding routing, and content formats — ensuring the system adapts to the learner, not the reverse.

🗂️
The Structured Thinker

Needs fixed layout, linear paths, step-by-step guidance, and calm sensory mode. Disrupted by unpredictable UI changes or too many simultaneous choices.

Fixed layout ON · Calm sensory mode ON · Step-by-step ON
🤝
The Supported Beginner

Benefits from guided mode, friendly errors, no timers, and visible progress. Builds confidence through scaffolded success.

Guided mode ON · Friendly errors ON · No timers ON
⚡
The Momentum Learner

Prefers short lessons, early hands-on practice, progress indicators, and clear next steps. Motivated by visible forward movement.

Short lessons ON · Progress indicators ON · Clear next step ON
🔭
The Deep Explorer

Wants detailed explanations, optional deep-dives, full error details, and free navigation. Resists over-scaffolding; needs depth and autonomy.

Detailed explanations ON · Free navigation ON · Save & resume ON

UX Decision

One question at a time

Single question per screen reduces cognitive overload. Full-width radio cards increase tap targets for motor accessibility. Progress bar provides orientation without pressure.

UX Decision

Affirming, not clinical

Result screen names the learner's type in positive, empowering language. No deficit framing. "Start Learning" CTA gives immediate forward momentum without overwhelming settings.

UX Decision

Persona as persistent identity

The "Deep Explorer" badge confirms the platform has adapted to the learner. Clean, low-distraction layout with status indicators instead of gamified urgency.

UX Decision

Structure without overwhelm

Progress shown as both bar and fraction. Lessons listed as discrete named steps. Back navigation always visible. Single current lesson highlighted at all times.

04 · Accessibility Design

Accessibility as architecture, not afterthought.

Six WCAG-aligned principles were embedded from the first design decision.

🎨

Colour-blind Friendly

High contrast design never relies on colour alone. All elements use text labels alongside colour cues. Blue on white exceeds WCAG AA ratios throughout.

WCAG 1.4.1 · 1.4.3
🧠

Reduced Cognitive Load

No autoplay, no default animation, no pop-ups. One primary action per screen. Information chunked to match working memory limits.

WCAG 2.2.2 · 3.2.1
📐

Large Touch Targets

All clickable elements minimum 44×44px. Radio options are full-width card rows — a deliberate choice for motor accessibility and touchscreen use.

WCAG 2.5.5 · 2.5.8
🔮

Predictable Navigation

Navigation structure never changes between pages. Back/Next always in the same position. Users always know where they are via section labels and progress indicators.

WCAG 3.2.3 · 3.2.4
✍️

Plain Language

All interface copy uses plain language. Personas are named with positive labels. Error messages are helpful, not alarming. No jargon in learner-facing text.

WCAG 3.1.5 · 3.3.2
⚙️

Adaptive Configuration

Learners can change their persona mode at any time via Settings without losing progress. Autonomy is preserved — no decision is permanent.

WCAG 3.2.2 · User Control

05 · Reflection

What I learned. What comes next.

"The most important design decision was to make the system adapt to the user — not ask the user to adapt to the system."
  • The persona coverage matrix was the most valuable design tool — it forced me to think about gaps before writing a single line of code.
  • With more time, I would conduct structured usability testing with actual Daglig Verksamhet participants to sharpen the onboarding routing logic.
  • I would extend the adaptive system to content format — a Deep Explorer should automatically see different explanation depth than a Momentum Learner.
  • This project raised a question I want to pursue at master's level: how do we evaluate whether an adaptive interface actually reduces cognitive load, versus simply feeling more comfortable?

Research Question for HCI Master's

"Can onboarding-driven persona systems measurably reduce task completion time and anxiety for neurodivergent learners in digital education environments — and how do we design the evaluation?"
